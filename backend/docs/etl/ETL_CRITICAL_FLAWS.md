# ETL Critical Flaws & Legacy Issues

This document identifies the fundamental architectural flaws in the current Graylinx ETL implementation that prevent it from being a "proper" industrial-grade system.

## 1. Logic Isolation (The "Black Box" Problem)
The most critical flaw is that the core transformation logic resides in **MySQL Stored Procedures** generated by a Javascript script.
- **Problem**: Stored procedures are "trapped" inside the database. They cannot be easily unit-tested, version-controlled, or debugged.
- **Impact**: If a calculation (like SPC or TR) is incorrect, finding the bug requires manually tracing through hundreds of lines of generated SQL.

## 2. Orchestration Blindness
The pipeline is triggered by an internal **MySQL Event Scheduler**.
- **Problem**: There is no visibility. There is no dashboard to see if an ETL run failed, why it failed, or how long it took.
- **Impact**: If the database hangs or a procedure crashes, the system fails silently. No alerts are sent, and data gaps are created without the team knowing.

## 3. Sequential Polling Bottlenecks
Data ingestion (`dataLoader.js`) is based on a **Sequential Polling** pattern.
- **Problem**: The system asks one device at a time for its data.
- **Impact**: As you add more sites and devices, the total time to poll everyone will eventually exceed the 15-minute window, causing the entire data ingestion process to collapse.

## 4. Tight Compute/Storage Coupling
The database is being used for both **Storage** and **Computation** (math).
- **Problem**: Heavy analytics runs use up the same CPU and memory needed for the actual Web App.
- **Impact**: The frontend may become slow or unresponsive during the 15-minute ETL bursts.
